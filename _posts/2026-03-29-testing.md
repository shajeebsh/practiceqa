### Q4: How will you implement re-running scripts logic?

**Your answer:** While loop, config limit, try-catch, increment on exception.

**Why it was weak:**

  * This retries **every** failure – including fatal errors (auth failure, app crash). That's wasteful and masks real bugs.
  * No distinction between **retryable vs non-retryable** exceptions.
  * No mention of **exponential backoff** or **logging**.
  * No mention of **framework-native retry** (Playwright, NUnit, Pytest already have this).



**What they expected (Lead level):**

> *"I first use **framework-native retry** where possible. For Playwright, `test.describe.configure({ retries: 2 })`. For NUnit, `[Retry(2)]`.
> 
> For custom retry logic, I distinguish between retryable and non-retryable failures:
>     
>     
>     int retryCount = 0;
>     int maxRetries = 2;
>     while (retryCount < maxRetries)
>     {
>         try
>         {
>             await page.ClickAsync("#submit");
>             break;
>         }
>         catch (StaleElementReferenceException) // retryable
>         {
>             retryCount++;
>             await Task.Delay(1000 * retryCount);
>         }
>         catch (AssertionException) // non-retryable – fail immediately
>         {
>             throw;
>         }
>     }
> 
> I also track retry rates. If a test retries more than 10% of the time, I flag it as flaky for investigation. On my Google project, this reduced false failure noise by **18 hours per month**."

**Key improvements:** Framework-native first, exception classification, backoff, metrics.
